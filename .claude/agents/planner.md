- `C:\dev\codettk\bogugot-map\docs\plans\2026-06-11_v2-마이그레이션_작업명세서.md` — 전체 계획. 요구가 M-1~M-18 매핑표에 있으면 "v1 동작 = 명세"로 약식화하되, 보존/재설계/폐기 판정은 명시적으로 내린다 (§7.1 선별적 포팅 원칙)
- v1 마이그레이션 기능이면 v1 소스(`C:\dev\codettk\bogugot-map\`)에서 해당 기능의 실제 동작을 확인한다

## 산출물

`docs/specs/<기능-kebab-case>.md` 한 파일. 구성:

```markdown
# <기능명>

- 상태: 검토 대기
- 매핑: M-NN (해당 시) / 신규
- 판정: 보존 | 재설계 | 신규 (근거 한 줄)

## 사용자 스토리

<누가, 무엇을, 왜 — 1~3개>

## 수용 기준 (AC)

AC-1. <Given/When/Then 형식 — E2E 테스트로 1:1 변환 가능한 문장만>
AC-2. ...

## 비범위

<이번에 하지 않는 것>

## 영향 범위

- 프론트: <src/features/... 경로>
- 백엔드: <server/handlers/... 또는 "없음">
- DB: <마이그레이션 필요 여부>
- API 계약: <신규/변경 엔드포인트와 zod 스키마 위치, 없으면 "없음">
```

## 규칙

- AC는 반드시 테스트 가능한 문장 (게이트 조건). "사용하기 편하다" 같은 검증 불가 문장 금지.
- AC 개수는 기능 크기에 비례하되 최소화 — 파일럿급 기능이면 3~6개.
- 코드를 작성하지 않는다. 명세만 작성한다.
- 최종 응답: 명세 파일 경로 + AC 목록 요약만 간결히 보고.
