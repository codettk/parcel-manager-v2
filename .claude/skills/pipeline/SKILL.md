# /pipeline — 기능 개발 파이프라인

입력: 기능 요구 한 줄 (`$ARGUMENTS`). 없으면 사용자에게 한 줄로 무엇을 만들지 묻는다.

너(메인 에이전트)는 오케스트레이터다. 직접 구현하지 말고 각 단계를 해당 서브에이전트(Agent tool, `subagent_type`에 에이전트 이름)에 위임하라. 각 단계는 명시적 산출물을 내고, 다음 단계는 그 산출물만 입력으로 받는다.

## 단계

### Stage 1 — 기획 (planner)

- `planner` 에이전트에 기능 요구를 전달 → `docs/specs/<기능>.md` 생성.
- **게이트 [사용자 승인 — 필수 중단점]**: 명세의 AC 목록을 사용자에게 보여주고 AskUserQuestion으로 승인/수정 요청을 받는다. 승인 전에는 절대 다음 단계로 가지 않는다. 수정 요청 시 planner에 반영시켜 재승인.

### Stage 2 — UI 디자인 ∥ API 계약 (병렬)

- `ui-designer` 에이전트: 명세 경로를 전달 → `design/bogugot.pen`에 화면 프레임. (Pencil 에디터에 파일이 열려 있어야 함 — 실패 시 사용자에게 열어달라고 요청)
- API 계약: 명세의 영향 범위에 API가 있으면 **오케스트레이터가 직접** `src/types/api/<기능>.ts`에 zod 스키마를 확정한다 (③·④의 공유 입력). API 없는 기능이면 생략.
- 게이트: ui-designer 보고에 토큰 위반·신규 UI 미신고가 없는지 확인. 신규 공통 컴포넌트가 필요하다고 보고되면 사용자에게 알리고 진행 여부 확인.

### Stage 3 — 구현 (frontend-dev ∥ backend-dev 병렬)

- 한 메시지에서 두 Agent 호출을 동시에 보낸다 (계약 기반 병렬). 백엔드 변경이 없는 기능이면 frontend-dev만.
- 전달물: 명세 경로, 디자인 프레임 ID, 계약 스키마 경로.
- 게이트: 각자 보고한 lint/typecheck/test 통과 여부 확인. 실패 보고 시 해당 에이전트에 SendMessage로 수정 지시 (최대 2회).

### Stage 4 — E2E (tester)

- `tester` 에이전트: 명세 경로 전달 → `tests/e2e/<기능>.spec.ts` 작성·실행.
- 게이트: 전체 green. 구현 결함 보고 시 Stage 3 해당 에이전트로 반려.

### Stage 5 — 검증 (verifier)

- `verifier` 에이전트: 명세 경로 + 변경 범위(기준 커밋) 전달 → 적대적 리뷰 보고.
- 게이트: blocking 0건. 발견 시 지목된 단계(③/④)로 반려 → 수정 → verifier 재실행. **반려 루프 최대 2회** — 초과 시 사용자에게 상황 보고 후 지시 대기.

### Stage 6 — 배포 준비 (deployer)

- `deployer` 에이전트 → 커밋(사용자 이름만, Claude 흔적 금지) + 마이그레이션 절차 + 릴리즈 노트 초안 + 문서 정합성 갱신.
- **게이트 [사용자 배포 승인 — 필수 중단점]**: 커밋 요약·릴리즈 노트를 보여주고 push/배포는 사용자가 결정. push를 대신 하지 않는다.

## 오케스트레이터 규칙

- 단계를 건너뛰지 않는다. 단, 명세의 영향 범위에 따라 해당 없는 단계(백엔드 없음 → backend-dev 생략, UI 없음 → ui-designer 생략)는 "해당 없음"으로 기록하고 넘어간다.
- 각 단계 종료 시 한 줄 상태 보고: `[Stage N] <산출물> — 게이트: 통과/반려`.
- 반려 시 어떤 게이트가 왜 실패했는지 다음 에이전트 프롬프트에 그대로 전달.
- 산출물 경로 규칙: 명세 `docs/specs/`, 계약 `src/types/api/`, 구현 `src/features/<domain>/`·`server/handlers/`, E2E `tests/e2e/`.
