# 계정·인증 기반 (소셜 로그인 + 세션 + 작업공간 멤버십)

- 상태: 승인됨 (결정 확정)
- 매핑: 신규 (필지 전국 전환 로드맵 슬라이스 2 — `docs/specs/pilji-roadmap.md`. M-1~M-18 매핑표 밖)
- 판정:
  - **인증/세션**: 신규 (현재 앱은 인증 부재 — 익명 공유 작업공간으로 부팅). Supabase Auth(GoTrue) 채택.
  - **익명 `clientId` 모델**: 보존 (Realtime 에코 가드 정체성 — 세션과 직교. 디바이스 식별자로 그대로 둔다. §결정 3 참조).
  - **작업공간 소유 모델**: 재설계 (익명 단일 공유 → 인증된 멤버 협업. 기존 tabs/parcel_settings 스키마는 비파괴 확장 — nullable `created_by`/멤버십 컬럼 신설, 기존 행 마이그레이션 불요).
  - **부팅 게이트**: 재설계 (슬라이스 1의 region-우선 게이트 위에 **로그인-우선 게이트**를 얹는다 — 로그인이 첫 관문, region 게이트는 로그인 성공 후로 후퇴).
  - **지도 코어·시트·동기화(M-1~16)**: 보존 (이번 슬라이스는 부팅 앞단에 세션 레이어만 얹는다).

## 배경 / 확정 사항 (반드시 후속 단계가 인지)

직전 슬라이스 `region-entry`(슬라이스 1)는 "네이티브 로그인·토큰 핸드오프는 설계만, 로그인 게이트 미강제"로 인증을 미뤘다. 이번 슬라이스가 그것을 받으며, **로그인을 앱 진입의 첫 강제 관문으로 승격**한다. 단:

- **이번 슬라이스는 웹 OAuth 실구현까지.** 네이티브 셸(Capacitor)·네이티브 소셜 로그인·푸시는 슬라이스 8. 따라서 네이티브→웹뷰 토큰 핸드오프는 **계약·인터페이스 설계 + 웹뷰 수신 코드 경로**까지만 만들고, 네이티브 발신부는 만들지 않는다 (스텁 경로로 E2E 가능하게).
- **제공자**: 카카오 웹 OAuth만 실구현(§결정 4 확정). Apple·휴대폰(전화 OTP)은 UI 자리·라우팅·계약만 두고 "준비 중"(비활성, 세션 미수립) 처리 — 실연동은 후속(특히 Apple은 슬라이스 8 스토어 등록 시점에 필요).
- **데이터 소유 모델은 이 슬라이스가 처음 건드리는 영역**이며, 5개 결정 포인트는 §결정 포인트에 **확정값**으로 못박는다.

## 부팅 순서 (확정 — 로그인 → region 게이트 → 지도)

```
앱 부팅
  │
  ├─ 세션 복원 시도 (영속 토큰 / 네이티브 핸드오프 수신)
  │
  ├─[세션 없음]──▶ 로그인 화면 (LoginView)  ◀── 앱 본문(지도·region 선택) 일체 미노출
  │                   │
  │                   └─ 카카오 OAuth 성공 ──▶ 세션 수립
  │
  └─[세션 있음]───────────────────────────────┐
                                              ▼
                          region 게이트 (슬라이스 1)
                              │
                              ├─[마지막 region 기록 없음]──▶ 지역 선택 화면 (RegionSelectView)
                              │                                  │
                              │                                  └─ region 선택 ──┐
                              │                                                    ▼
                              └─[마지막 region 기록 있음]──────────────────────▶ 지도 (MapCanvas)
```

핵심: **로그인이 region 게이트보다 앞선다.** 비로그인 상태에서는 region 선택 화면조차 노출하지 않는다(로그인 화면만). 로그인 성공 후에야 슬라이스 1의 region 게이트 로직(`pilji_v2_active_region` 영속 키 분기)이 작동한다. 로그인됐고 마지막 region 기록이 있으면 region 선택을 건너뛰고 지도로 직행한다.

## 사용자 스토리

1. 새 사용자로서, 앱을 열면 곧바로 로그인 화면을 만나 카카오로 한 번에 로그인해 내 신원이 붙은 계정을 만들고 싶다 (재방문 시 같은 작업 이력에 접근).
2. 돌아온 사용자로서, 앱을 다시 열었을 때 다시 로그인하지 않고 이전 세션이 유지된 채 곧바로 (마지막 region이 있으면) 지도로 복귀하길 원한다 (세션 + region 영속).
3. 로그인한 사용자로서, 같은 작업공간에 함께 들어온 다른 멤버와 동일 지도를 협업해 칠하되, 내 변경에 내 신원이 남길 원한다.
4. 로그인한 사용자로서, "내 정보" 화면에서 내 계정(닉네임·연결된 로그인 수단)을 확인하고 로그아웃할 수 있길 원한다.
5. 로그인이 실패(취소·네트워크·핸드오프 오류)하면, 앱이 멈추지 않고 명확한 에러 화면과 재시도 경로를 보고 싶다.

## 수용 기준 (AC)

### 로그인 게이트 — 진입 강제 (§결정 1 확정: 로그인 필수)

AC-1. (Given 비로그인 상태로 앱이 부팅을 마친 상태) When 화면이 렌더된다 Then 로그인 화면(LoginView)만 표시되고, 지도 캔버스·탭바·지역 선택 화면·지역칩 등 앱 본문 요소가 일체 노출되지 않는다.

AC-2. (Given AC-1의 로그인 화면) When 사용자가 "카카오로 시작하기"를 탭하고 OAuth 콜백이 성공으로 돌아온다 Then 세션이 수립되고 로그인 화면이 닫히며, 곧바로 region 게이트(지역 선택 또는 마지막 region 지도)로 진입한다.

AC-3. (Given 카카오 로그인에 성공했고 `pilji_v2_active_region`에 마지막 region 기록이 **없는** 상태) When 로그인이 완료된다 Then 지역 선택 화면(RegionSelectView)이 표시된다 (지도 직행 아님 — region 게이트가 로그인 다음 관문).

AC-4. (Given 카카오 로그인에 성공했고 마지막 region 기록이 **있는** 상태) When 로그인이 완료된다 Then 지역 선택 화면을 거치지 않고 곧바로 마지막 region의 지도로 진입한다 (로그인→region영속→지도 직행).

### 세션 영속 / 로그아웃

AC-5. (Given AC-2로 로그인되어 세션이 영속되고 마지막 region 기록이 있는 상태) When 사용자가 앱을 새로고침한다 Then 로그인 화면을 다시 거치지 않고 세션이 복원되어 곧바로 지도로 복귀한다 (세션 토큰 영속).

AC-6. (Given 로그인된 상태에서 "내 정보" 화면) When 사용자가 "로그아웃"을 탭한다 Then 세션이 파기되고 화면이 로그인 화면(LoginView)으로 복귀하며, 지도·region 선택 등 앱 본문이 다시 미노출 상태가 된다.

### 로그인 실패 / 핸드오프 에러

AC-7. (Given 로그인 화면) When OAuth 흐름이 사용자 취소 또는 제공자 오류로 실패 콜백을 반환한다 Then 디자인된 핸드오프 에러 화면(모바일 ㊹)이 표시되고 "다시 시도" 진입점이 제공되며, 앱은 크래시 없이 비로그인(로그인 화면) 상태로 복귀한다.

AC-8. (Given 카카오 외 제공자 버튼 — Apple·휴대폰) When 사용자가 해당 버튼을 탭한다 Then "준비 중" 안내가 표시되고 세션은 수립되지 않으며 앱 본문에 진입하지 않는다 (이번 슬라이스 실연동 비범위).

### 계정 신원이 데이터에 반영 (멤버십 모델 — §결정 2 확정: 작업공간 멤버십)

AC-9. (Given 로그인된 사용자가 필지를 편집해 저장한 상태) When 해당 변경이 서버에 기록된다 Then 행의 신원 식별자(`updated_by`/`created_by`)가 인증된 사용자 식별자(user_id) 기반 값으로 기록된다 (익명 디바이스 `clientId`가 신원 컬럼을 덮지 않는다).

AC-10. (Given 같은 작업공간에 로그인한 다른 사용자가 동시에 접속) When 한 사용자가 필지를 변경한다 Then 다른 사용자의 화면에 Realtime으로 변경이 반영되고(기존 M-6 공유 협업 동작 보존), 자기 디바이스 에코는 무시된다 (clientId 에코 가드 회귀 없음 — §결정 3).

### clientId 에코 가드 ↔ 세션 신원 분리 (§결정 3 확정: 분리)

AC-11. (Given 같은 user_id로 두 개의 서로 다른 디바이스에서 동시 접속한 상태) When 디바이스 A가 필지를 변경한다 Then 디바이스 B에는 Realtime 변경이 반영되고(같은 user_id여도 에코로 무시되지 않음), 디바이스 A 자신에게만 에코 가드가 적용된다 (에코 가드는 user_id가 아니라 디바이스 `clientId` 단위로 동작).

### 서버 핸들러 인증 강제 (§결정 5 확정: RLS 미도입, 핸들러 강제)

AC-12. (Given 유효한 세션 토큰 없이 mutate API를 직접 호출) When 요청이 서버 핸들러에 도달한다 Then 핸들러가 401을 반환하고 어떤 행도 기록되지 않는다 (RLS가 아니라 핸들러 레이어에서 인증 강제 — `0002_disable_rls` 유지).

### 네이티브 토큰 핸드오프 (계약만 — 실구현 비범위)

AC-13. (Given 앱이 네이티브 웹뷰 컨텍스트로 부팅되어 `AuthBridge`가 토큰 페이로드를 주입하는 경로) When 유효한 핸드오프 토큰이 웹뷰 수신 인터페이스로 전달된다 Then 웹은 그 토큰으로 세션을 수립하고 로그인 화면을 건너뛴다 (수신 경로·계약을 스텁/모킹으로 검증 — 네이티브 발신부는 슬라이스 8).

AC-14. (Given AC-13 경로) When 핸드오프 토큰이 누락/만료/형식 오류다 Then 핸드오프 에러 화면(㊹)으로 폴백하고 웹 카카오 OAuth 경로로 재시도할 수 있다.

## 비범위

- PRO 구독·freemium 게이팅·페이월·IAP (슬라이스 6).
- 전국 지적도 다중 region 데이터 파이프라인 (슬라이스 3) — region 모델은 슬라이스 1 클라이언트 메타 그대로.
- 실제 네이티브 셸(Capacitor) 빌드·네이티브 소셜 로그인 발신·푸시·스토어 등록 (슬라이스 8). 본 슬라이스는 웹뷰 토큰 **수신 계약**까지.
- Apple·휴대폰(전화 OTP) 제공자 실연동 (자리·계약만, "준비 중").
- RLS 정책 재도입 (멤버십은 서버 핸들러 레이어 강제 — §RLS 메모).
- 작업공간 초대/멤버 관리 UI, 권한 등급(소유자/편집자/뷰어) (후속 — 이번엔 "로그인하면 기존 공유 작업공간의 멤버"가 디폴트).
- 사용자별 데이터 격리(소유 모델 b) — §결정 2에서 멤버십(a)으로 확정, 격리 모델 폐기.
- 익명 열람 / 익명 쓰기 게이트 (구 권장안 — §결정 1에서 로그인 필수로 확정, 익명 경로 전면 폐기).

## 영향 범위

- 프론트:
  - 신규 `src/features/auth/` — `LoginView`(풀스크린 진입 게이트, 카카오 CTA + Apple·휴대폰 "준비 중"), `HandoffErrorView`(㊹), `AccountSheet`(내 정보 — 계정·연결 수단·로그아웃), `authBridge.ts`(네이티브 토큰 수신 계약 인터페이스 + 웹 폴백), `useSession.ts`(세션 구독 훅).
  - `src/lib/auth.ts` (신규) — Supabase Auth 클라이언트 래퍼: `signInWithKakao()`·`signOut()`·`getSession()`·`onAuthStateChange()`·`setSessionFromHandoff(token)`. 단일 supabase 클라이언트 인스턴스 공유(현재 realtime.ts가 매번 `createClient` 하는 부분과 통합 검토 — 인증 세션과 Realtime이 같은 클라이언트를 쓰면 토큰이 Realtime 구독에도 적용된다).
  - `src/stores/` — 세션 상태 슬라이스 (신규 `auth.ts`): `session`(null=비로그인)·`user`·`authStatus`(loading/authenticated/error). **부팅 게이트 우선순위**: `authStatus !== authenticated` 이면 region 게이트·지도·`boot()` 일체 차단하고 LoginView만 렌더. 인증 후에야 슬라이스 1의 region 게이트(`pilji_v2_active_region`) 분기 → `boot()`.
  - `src/lib/api.ts` — mutate 시 인증 토큰 첨부(Authorization 헤더). `clientId`는 그대로 유지(에코 가드, 신원과 분리 — §결정 3).
  - `src/App.tsx` — 부팅 시퀀스 최상단에 세션 복원/로그인 게이트 분기 추가(로그인→region→지도 순서), "내 정보" 진입점(NavDrawer), 로그인/핸드오프 에러 뷰 분기. 비로그인 시 region 게이트·지도·탭바 렌더 경로 진입 자체를 막는다.
  - `src/features/tab/NavDrawer.tsx` — "내 정보"·로그아웃 항목 추가.
  - `index.html`/`vite` — OAuth 콜백 라우트(`/auth/callback`) 처리 경로.
- 백엔드: **필요 (backend-dev 슬라이스)**.
  - `server/handlers/` — 모든 mutate 핸들러에 세션 검증 헬퍼 적용(Authorization Bearer → user 검증 → 신원 컬럼=user_id 주입, 무토큰 401). 신규 `server/handlers/auth.ts`(세션 검증 헬퍼·내 정보 조회). `dispatch`가 토큰을 런타임 비의존 핸들러 컨텍스트로 전달(req/res 직접 접근 금지 원칙 유지).
  - `server/routes.ts` — `GET /api/me`(내 정보) 등 신규 라우트.
  - Supabase Auth 제공자 설정(카카오 OAuth 앱·리다이렉트 URL) — 인프라 설정(코드 외 + 환경변수).
- DB: **마이그레이션 필요** (신규 `supabase/migrations/0003_auth_membership.sql`).
  - Supabase `auth.users`는 기본 제공 — `public.profiles`(닉네임·연결 제공자) 보조 테이블은 선택.
  - 멤버십 모델(§결정 2 확정 — 작업공간 멤버십): `tabs`/`parcel_settings`/`parcel_groups`/`color_labels`에 **nullable `created_by uuid REFERENCES auth.users`** 추가(기존 익명 행 비파괴 — NULL 허용). 에코 가드용 식별자와 신원 식별자 분리(§결정 3): 기존 `updated_by`(에코 가드 clientId)는 의미 그대로 두고, 신원은 별도 `created_by`/`updated_by_user` 계열 컬럼으로 분리해 에코 가드(`updated_by === clientId` 비교) 회귀 방지. 작업공간↔사용자 매핑이 필요하면 `workspace_members` 테이블 신설(nullable·비파괴).
  - **기존 행 데이터 마이그레이션 불요** (전부 nullable 신설 — 기존 익명 협업 행 그대로 유효).
  - RLS: 이번 슬라이스 미도입(`0002_disable_rls` 유지) — §RLS 메모.
- API 계약: **신규** — `src/types/api/auth.ts`.
  - `meResponseSchema`(user_id·닉네임·이메일·연결 제공자), `handoffTokenSchema`(네이티브 핸드오프 토큰 페이로드), 401 인증 실패 형태(기존 `errorResponseSchema` 재사용).
  - 기존 mutate 계약(tabState·colors·tabs 등)은 바디 무변경 — 인증은 헤더 레이어. `clientId`는 계약에 그대로 유지(신원과 직교).

## 결정 포인트 (확정됨)

> 5개 결정 포인트 전부 사용자 승인으로 확정. 본 AC·영향 범위는 아래 확정값에 1:1 종속한다.

**결정 1 — 인증 강제 범위 / region 게이트와의 순서 → 확정: 로그인 필수 (1b)**
- 앱 진입 = 로그인 게이트 우선. 부팅 순서 **로그인 → region 선택 게이트(슬라이스1) → 지도**. 비로그인 상태에서는 지도·지역 선택을 포함해 앱 본문 일체 미노출(로그인 화면만).
- 권장했던 "익명 열람 + 쓰기 시 로그인(1a)"은 **폐기**. 구 AC-6·7·8(익명 열람/쓰기 게이트)은 진입 게이트 AC(AC-1~4)로 대체.
- 영향: AC-1~6, App 부팅 분기, region 게이트가 로그인 다음으로 후퇴.

**결정 2 — 데이터 소유 / 멤버십 모델 → 확정: 작업공간 멤버십 (2a)**
- 기존 공유 협업 유지 + 행에 신원 부여. 로그인 사용자는 디폴트로 기존 공유 작업공간의 멤버. M-6 Realtime 협업·기존 tabs 구조 보존, **nullable 컬럼 신설로 비파괴 마이그레이션**.
- 사용자별 데이터 격리(2b)는 **폐기**.
- 영향: AC-9·10, DB nullable `created_by`/`workspace_members` 신설.

**결정 3 — `clientId` 에코 가드와 세션의 공존 → 확정: 분리**
- `clientId`(디바이스 UUID)는 에코 가드 정체성으로 **보존**, 신원(user_id)은 **별도 컬럼/필드로 분리** 기록. `updated_by`(에코 가드용)를 user_id로 덮지 않는다 — 같은 사용자가 두 디바이스로 접속해도 에코 가드가 디바이스 단위로 정확.
- 영향: AC-9·11, DB 신원 컬럼을 에코 가드 컬럼과 별도로 신설.

**결정 4 — 제공자 범위 → 확정: 카카오만 실구현**
- 카카오 웹 OAuth만 실구현. Apple·휴대폰은 UI 자리·계약·"준비 중"(세션 미수립)만. Apple 실연동은 슬라이스 8 iOS 스토어 등록 시점.
- 영향: AC-2·8.

**결정 5 — RLS 재도입 시점 → 확정: 미도입 (이번 슬라이스 제외)**
- 멤버십 강제는 서버 핸들러(service_role) 레이어에서. 현재 `0002_disable_rls` 유지. RLS + auth.uid() 정책 + Realtime authenticated 전환은 별도 후속 슬라이스.
- 영향: AC-12, §RLS 메모.

## RLS 메모 (후속)

현재 `0002_disable_rls`로 모든 테이블 RLS OFF + anon 키 직접 접근 가능(v1 posture). 멤버십(결정2)이 자리잡고 사용자별 권한이 의미를 가지면, RLS + auth.uid() 기반 정책 + Realtime 구독을 authenticated 토큰으로 전환하는 별도 슬라이스가 필요하다. 이번 슬라이스는 세션·신원만 도입하고 권한 강제는 서버 핸들러 레이어로 한정한다(AC-12).
